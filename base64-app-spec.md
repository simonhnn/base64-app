# Base64変換ウェブアプリ — 仕様書（確定版）

作成日: 2026-07-22 ／ ステータス: 確定

---

## 1. 概要

テキストの Base64 エンコード / デコード（相互変換）を行うウェブアプリ。変換処理はすべてクライアント（ブラウザ）側で完結させ、サーバーには「変換データの収集ログ」だけを送信する。ホスティングと収集バックエンドは Cloudflare の無料枠で構築する。開発は 2 フェーズに分け、フェーズ 1 で無料・無広告のアプリを公開し、フェーズ 2 で広告・独自ドメインによる収益化を追加する（詳細は「7. 開発フェーズ」）。

### 確定した設計方針

- 変換はクライアント側で完結（サーバー往復ゼロ・高速・プライバシー配慮）
- フロントは素の HTML/CSS/JS 単一構成。フレームワークを使わない（描画速度最優先）
- UI はシンプル。入力欄・出力欄・モード切替・コピー・クリアの最小要素のみ
- サイト本体は静的、収集ログの保存だけ動的処理（Pages Functions + D1）に分離
- サーバー管理不要（サーバーレス）
- 保存する変換データは、変換前後を問わず **Base64 ではない側（平文側）** に統一
- 個人〜中規模なら実質月額 0 円

## 2. 機能要件

### 2.1 変換機能（コア）

| 項目 | 内容 |
|---|---|
| モード切替 | エンコード（テキスト→Base64）/ デコード（Base64→テキスト）を切替 |
| 入力上限 | 最大 10,000 文字。超過分は入力を制限し、上限に達した旨を表示 |
| テキスト → Base64 | UTF-8 対応（日本語などマルチバイト含む）。`TextEncoder` → `btoa` |
| Base64 → テキスト | `atob` → `TextDecoder`。不正な入力はエラー表示 |
| リアルタイム変換 | 入力に応じて即時変換（ボタン不要）。描画を軽く保つ |
| コピー | 出力をワンクリックでクリップボードにコピー |
| クリア | 入力・出力のリセット |

### 2.2 収集機能

| 項目 | 内容 |
|---|---|
| 保存対象 | **Base64 ではない側（平文側）のみ**。エンコード時は入力、デコード時は出力を保存 |
| 付随情報 | 変換方向（encode / decode）、日時 |
| 送信タイミング | 入力確定後（デバウンス 1.5 秒）またはコピー実行時に 1 回だけ POST。キー入力ごとには送らない（書き込み量と断片化を抑制） |
| 送信先 | 自サイトの `/api/log`（Pages Functions）経由で D1 に保存。第三者提供なし |

### 2.3 非機能要件

| 項目 | 内容 |
|---|---|
| 処理速度 | 変換は瞬時（クライアント）、配信は CDN エッジキャッシュで高速 |
| 描画速度 | 初期表示・再描画を最優先。フレームワーク不使用、依存を最小化。フォント・画像・スクリプトを最小化 |
| UI | シンプルさ優先。最小要素のみ、過剰な装飾・アニメーションを持たない |
| プライバシー | 変換自体はサーバーに送らない。保存は平文側のみで第三者提供なし |
| セキュリティ | 収集エンドポイントにレート制限、HTTPS 必須 |
| コスト | 個人〜中規模で無料枠内 |

### 2.4 広告（フェーズ 2）

- HTML に広告タグ（AdSense 等）を埋め込むだけで動作。サーバー側の追加インストール不要
- 審査対策として、使い方の説明ページ等のコンテンツを用意
- 広告設置・独自ドメイン取得はフェーズ 2 で着手（フェーズ 1 は無広告）

## 3. 保存データ仕様（確定）

変換前後のどちらであっても、保存するのは **Base64 ではない側（平文側）** で統一する。

| 変換方向 | 入力 | 出力 | 保存する側 |
|---|---|---|---|
| エンコード | 平文 | Base64 | 入力（平文） |
| デコード | Base64 | 平文 | 出力（平文） |

理由：平文側は分析価値が高く、Base64（約 1.33 倍に膨張）より容量効率も良いため。

> プライバシー注記：平文側は氏名・パスワード等が直接読める形で保存され得る。取得は最小限にとどめ、プライバシーポリシーで利用目的・第三者提供なしを明示する。将来的に機微情報の混入が懸念される場合は、平文の保存をやめ集計値（文字数等）のみに切り替える運用も可能な設計とする。

## 4. Cloudflare Pages 構成

### 4.1 全体構成図

```
[ユーザーのブラウザ]
        │
        │ ① 静的アセット取得（HTML/CSS/JS）
        │    → Cloudflare エッジCDNからキャッシュ配信（帯域無制限）
        ▼
[Cloudflare Pages（静的ホスティング）]
        │
        │ ② 変換はブラウザ内JSで完結（サーバー往復なし）
        │
        │ ③ 平文側のログを POST（fetch・デバウンス/コピー時）
        ▼
[Pages Functions（Workers）— /api/log]
        │
        │ ④ INSERT（平文側 + 方向 + 日時）
        ▼
[D1（SQLite互換DB）]
```

### 4.2 各コンポーネント（確定）

| 層 | 採用 | 役割 |
|---|---|---|
| 静的ホスティング | **Cloudflare Pages** | HTML/CSS/JS をエッジ配信。帯域無制限 |
| 変換処理 | **クライアント JS（素の JS）** | Base64 エンコード / デコード。サーバー不要 |
| 収集バックエンド | **Pages Functions（Workers）** | `/api/log` で POST を受け、D1 へ INSERT |
| データベース | **D1（SQLite互換）** | 平文側ログ・方向・日時を保存 |
| ドメイン | フェーズ 1: `*.pages.dev` ／ フェーズ 2: 独自ドメイン | — |

### 4.3 ディレクトリ構成（確定）

```
/
├── public/              # 静的アセット（Pagesが配信）
│   ├── index.html       # UI（入力/出力/モード切替/コピー/クリア）
│   ├── style.css        # 最小限のスタイル
│   └── app.js           # 変換処理 + 収集POST（デバウンス）
├── functions/
│   └── api/
│       └── log.js       # POST受け口 → D1へINSERT・レート制限
├── schema.sql           # D1テーブル定義
└── wrangler.toml        # D1バインディング等の設定
```

### 4.4 データフロー

1. ユーザーがサイトにアクセス → Pages が静的ファイルをエッジから配信
2. ユーザーが入力 → ブラウザ内 JS がリアルタイム変換（サーバーに送らない）
3. 入力確定（デバウンス 1.5 秒）またはコピー時 → 平文側 + 方向を `/api/log` へ POST
4. Pages Functions がレート制限のうえ D1 に 1 行 INSERT

### 4.5 D1 テーブル定義（確定）

```sql
CREATE TABLE conversion_logs (
  id         INTEGER PRIMARY KEY AUTOINCREMENT,
  direction  TEXT NOT NULL,          -- 'encode' or 'decode'
  plaintext  TEXT,                   -- Base64ではない側（平文）。encode時=入力, decode時=出力
  created_at TEXT NOT NULL DEFAULT (datetime('now'))
);
```

## 5. コスト

### 5.1 無料枠（月額 0 円の範囲・2026年時点）

| リソース | 無料枠 | この用途での消費 |
|---|---|---|
| Pages 帯域 | 無制限 | 気にしなくてよい |
| Pages ビルド | 月 500 回 | 個人開発なら余裕 |
| Workers 呼び出し | 1日 10万回 | 収集APIに使用 |
| D1 読み取り | 1日 500万行 | 余裕 |
| **D1 書き込み（INSERT）** | **1日 10万行** | 実質の上限 |
| D1 ストレージ | 5GB | ログなら相当貯められる |

### 5.2 超過時

| ケース | コスト |
|---|---|
| 通常の個人利用 | 月 0 円 |
| 無料枠を超えた | Workers Paid 月 5 ドル + D1書き込み 100万行あたり約 1 ドル |
| 独自ドメイン使用（フェーズ 2） | 年約 10.44 ドル（.com）〜 15 ドル |

無料枠を超えても課金ではなく処理が一時停止する設計のため、意図しない高額請求が起きにくい。

## 6. セキュリティ・プライバシー上の注意点

- 保存対象は平文側のみ。平文には氏名・メール・パスワード・APIキー等が混入し得るため、取得は最小限にとどめる
- 収集エンドポイント（`/api/log`）に **レート制限** をかけ、大量の不正 INSERT を防ぐ
- 入力上限（10,000 文字）はクライアントで制限しつつ、`/api/log` 側でも超過データを拒否（クライアント制限は回避され得るため）
- Pages Functions 内で INSERT を実行するため、DB 認証情報はブラウザに露出しない
- HTTPS は Cloudflare が自動で提供
- プライバシーポリシーを設置し、取得情報・利用目的・第三者提供なし・問い合わせ先を明示（個人情報保護法・電気通信事業法 外部送信規律への最低限の対応）
- フェーズ 2 で広告（Cookie）を導入する場合、プライバシーポリシーを更新し、EU 向けアクセスがあれば同意バナー（CMP）を追加

## 7. 開発フェーズ

### フェーズ 1：アプリ本体の公開（無料・無広告）

目的：テキスト相互変換ツールを動く状態で公開し、利用状況を把握できるようにする。

1. 静的フロント（HTML/CSS/JS）でテキスト相互変換機能を実装（リアルタイム変換・コピー・クリア）
2. `*.pages.dev`（無料サブドメイン）にデプロイして公開
3. Pages Functions で `/api/log` を作成、D1 バインディング設定（レート制限含む）
4. D1 にテーブル作成（`schema.sql`）
5. フロントから平文側の収集 POST を接続（デバウンス）
6. プライバシーポリシー・使い方の説明ページを用意

この時点でのコストは 0 円（`*.pages.dev` + Cloudflare 無料枠のみ）。

### フェーズ 2：収益化（広告・独自ドメイン）

目的：フェーズ 1 で公開したアプリに収益化を追加する。

1. 独自ドメインを取得し Cloudflare Pages に紐づけ（.com 年約 10.44 ドル 等）
2. 審査対策として説明系ページを拡充（使い方・解説、運営者情報、プライバシーポリシー、お問い合わせ）
3. Google AdSense アカウント開設 → `<head>` に広告コードを設置
4. 審査をリクエスト（完了まで 4 週間前後を見込む）
5. 合格後、広告ユニットを配置
6. 広告（Cookie）利用に合わせてプライバシーポリシーを更新。EU 向けアクセスがあれば同意バナー（CMP）を追加
7. （AdSense 不合格時の保険）忍者 AdMax・i-mobile 等の代替ネットワーク、または説明ページへのアフィリエイトを検討

この時点で追加されるコストは独自ドメイン代（年約 10〜15 ドル）のみ。ホスティングは引き続き無料枠内。

> 広告・独自ドメインはフェーズ 2 まで着手しない。フェーズ 1 は `*.pages.dev` のまま無広告で進める。
